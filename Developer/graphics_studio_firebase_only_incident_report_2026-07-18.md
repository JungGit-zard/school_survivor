## 1. 결론

이번 사고는 사용자의 “그래픽 스타지오 값은 Firebase에 모두 저장하고 그 값만 사용한다”는 명령을 구현 단계에서 “로컬 값을 원본으로 유지하면서 Firebase와 동기화한다”로 잘못 바꿔 해석한 것이 시작점이다.

그 결과 기존 브라우저 로컬 저장값, 기본 모델, Firebase 원격값이 동시에 존재하는 다중 원본 구조가 만들어졌다. 앱은 Firebase 확인 전에 로컬 또는 기본 상태로 먼저 시작할 수 있었고, 원격 데이터가 없을 때는 로컬 상태를 Firebase에 올렸으며, 연결 실패 때도 로컬 상태를 유지하도록 작성됐다. 이 구조에서는 구형 모델의 재노출과 계정 간 상태 오염이 필연적으로 발생할 수 있다.

오늘 긴급 수정 과정에서는 코드가 바뀐 뒤에도 5173 개발 서버가 이전 변환 결과를 계속 제공했다. 파일 수정만 확인하고 실제 서버 응답을 즉시 확인하지 않아 사용자는 다시 구형 화면을 보게 됐다. 이후 실제 서버 응답과 브라우저 본문을 다시 확인했고, 전체 앱을 막던 오류 게이트를 제거한 뒤 기존 Google 로그인 화면이 게임과 Studio 양쪽에서 표시되는 것을 확인했다.

## 3. 실제로 잘못 작성된 구조

### 3.1 기존 로컬 저장 구조를 제거하지 않고 Firebase를 덧붙였다

그래픽 스타지오 최초 구현에는 이미 브라우저 로컬 저장 구조가 있었다. Firebase 작업 시 이 구조를 폐기해야 했지만, 기존 로더와 세이버를 그대로 원본 계층으로 유지하고 Firebase를 그 위에 추가했다.

잘못된 당시 구현은 다음 동작을 포함했다.

1. 그래픽 튜닝, 효과음 튜닝, 보스 카드 미리보기, 데칼, 맵 프랍을 먼저 로컬 저장소에서 읽음.
2. Firebase 원격 경로가 비어 있으면 현재 로컬 상태를 원격에 최초 업로드함.
3. Firebase 읽기 실패 시 로컬 상태를 유지함.
4. Firebase 원격값을 적용할 때도 원격값을 로컬 저장 API에 다시 기록함.
5. 다른 탭과의 동기화도 브라우저 로컬 저장 이벤트를 사용함.

이 구조는 Firebase가 단일 원본이 아니라 로컬 상태의 백업 및 탭 동기화 보조 수단이 되게 만들었다.

### 3.2 잘못된 설계를 테스트가 정답으로 고정했다

당시 테스트에는 다음 동작이 성공 조건으로 들어갔다.

- 원격 경로가 비어 있으면 로컬 데이터를 원격에 seed한다.
- 원격 읽기가 실패해도 모든 로컬 데이터가 그대로 남아야 한다.
- Firebase hydrate는 원격값을 로컬 저장 API를 통해 적용한다.

즉, 테스트는 사용자 명령을 검증하지 않고 반대로 위반 동작을 회귀 방지 계약으로 만들었다. 그래서 테스트가 통과해도 제품 요구사항은 실패하는 상태가 됐다.

### 3.3 앱 시작 순서가 잘못됐다

타이틀과 게임이 Firebase hydrate 성공 전에 렌더될 수 있었다. 컴포넌트가 먼저 마운트되면 모델 로더는 로컬 저장값 또는 코드 기본값을 사용한다. 그 뒤 Firebase가 늦게 도착해 값을 바꾸더라도 사용자는 최소 한 프레임 이상 구형 모델을 볼 수 있고, 일부 컴포넌트가 초기값을 캡처하면 구형 상태가 계속 남을 수 있었다.

### 3.4 오늘은 개발 서버의 구형 변환 결과가 추가로 노출됐다

긴급 수정 파일은 디스크에 반영됐지만, 실행 중이던 Vite 개발 서버가 이전 `main.jsx`를 계속 제공했다. 실제 HTTP 응답에는 새 차단 코드가 아니라 기존 정적 App import가 남아 있었다.

잘못된 대응 순서:

1. 파일 diff와 테스트부터 확인함.
2. 사용자가 보고 있는 5173 서버의 실제 응답을 먼저 확인하지 않음.
3. 결과적으로 디스크 코드는 최신인데 브라우저는 구형 코드를 받는 상태를 방치함.

수정 후에는 서버를 재시작하고 실제 응답에서 다음 항목을 확인했다.

- 부팅 첫 단계의 Studio 로컬 저장 차단기 설치
- 게임 타이틀과 기존 Google 로그인 UI 정상 렌더
- Firebase 성공 전 Studio 의존 3D 화면만 지연
- 인증 후 `remote-applied` 상태에서 Studio와 Studio 의존 화면 개방

## 6. 현재 수정 내용

### 6.1 전체 앱 게이트 제거와 기존 로그인 흐름 복구

`src/App.jsx`의 전체 앱 차단용 `FirebaseStudioGate`와 치명적 오류 전면 화면을 제거했다.

- 게임 타이틀과 기존 `GoogleAccountPanel`은 로그인 전에도 정상 표시
- 로그인 후 인증 사용자를 기준으로 Studio 원격 스냅샷 hydrate
- 타이틀 3D, 로비 보스 카드, Graphics Studio처럼 Studio 값에 의존하는 화면만 `remote-applied`까지 지연
- 게임 시작 버튼은 로그인 직후 원격 hydrate 완료를 기다린 다음 로비로 이동
- 원격 실패 시 로컬값 또는 코드 기본값을 Studio 원본으로 대체하지 않음

### 6.2 Studio 데이터는 Firebase hydrate 후 런타임 메모리만 사용

`src/lib/studioRuntimeState.js`를 추가했다.

- 초기 상태는 `blocked`
- Firebase 스냅샷이 정상 검증된 뒤에만 `ready`
- 준비 전 Studio 데이터 읽기·쓰기 시 `FirebaseStudioNotHydratedError`
- 데이터셋은 그래픽 튜닝, 효과음 튜닝, 보스 카드 미리보기, 데칼, 맵 프랍 다섯 종류만 취급

런타임 메모리는 영구 저장소가 아니다. 화면에서 편집 중인 값을 잠시 들고 있는 작업 공간이며, 영구 원본은 Firebase 한 곳뿐이다.

### 6.3 Studio 로컬 저장 접근을 치명적 오류로 처리

`src/lib/studioLocalStorageGuard.js`를 추가하고 `src/main.jsx`에서 App import보다 먼저 설치한다.

다음 Studio 계열 키에 대해 브라우저 로컬 저장소의 읽기, 쓰기, 삭제를 시도하면:

1. 전체 화면 `alertdialog`를 표시함.
2. 금지된 연산과 키를 표시함.
3. 즉시 `FatalStudioLocalStorageError`를 던져 실행을 중단함.

로컬 저장소 전체 초기화도 Studio 상태를 함께 지울 수 있으므로 차단한다.

이 차단 범위는 그래픽 스타지오 데이터다. 게임 진행도, 사용자 설정 등 별도 시스템의 로컬 저장 구조는 이번 사고 범위에서 변경하지 않았다.

### 6.4 창 간 동기화도 Firebase 재조회로 변경

Studio가 게임 창으로 데이터 payload를 직접 보내지 않는다.

1. Studio가 Firebase 저장을 완료함.
2. 상대 창에는 Firebase를 다시 조회하라는 신호만 보냄.
3. 상대 창은 인증된 사용자 경로에서 Firebase 스냅샷을 다시 hydrate함.

따라서 창 간 전달 데이터가 별도의 원본이 되지 않는다.

### 6.5 개발 서버 전달 상태 재검증

기존 5173 서버 프로세스를 종료하고 같은 프로젝트·브랜치에서 다시 시작했다.

실제 HTTP 200 응답과 브라우저 본문으로 게임 타이틀, 기존 Google 로그인, Graphics Studio 로그인 안내가 제공되는 것을 확인했다. 디스크 파일 확인만으로 완료 처리하지 않았다.

### 6.6 Firebase 키 제약 결함 수정

최초 원격 저장 시 Studio 파츠 경로에 포함된 점 문자가 Realtime Database 객체 키 규칙에 걸려 쓰기가 거부되는 결함을 발견했다.

- 다섯 데이터셋을 각각 JSON 문자열로 감싸 Firebase에 저장
- hydrate 시 문자열을 다시 정확한 객체와 원래 파츠 키로 복원
- 점이 포함된 실제 파츠 경로의 왕복 테스트 추가
- 데이터셋별 문자열 형식과 최대 길이를 검증하도록 운영 규칙 갱신 및 재배포

- 원격 최초 스냅샷 저장: 성공
- 원격 재조회와 승인 원본 해시 비교: 일치
- 현재 개발 서버: `localhost:5173` 게임·Studio 모두 HTTP 200
- 실제 브라우저 본문: 기존 Google 로그인 UI와 두 경로 정상 표시

검증 대상:

- Firebase Studio 원격 hydrate 및 저장
- 로컬 저장 접근 치명적 차단
- Graphics Studio 편집·적용·게임 연결
- 그래픽 튜닝 런타임
- 효과음 튜닝 런타임
- 맵 프랍 배치 런타임
- App Firebase 게이트와 모바일 입력 회귀

테스트 중 React `act` 환경 경고는 남아 있지만 실패는 없었다. 이 경고는 이번 저장 원본 사고의 원인이 아니며 73개 검증은 모두 통과했다.

## 8. 재발 방지 규칙

앞으로 Graphics Studio 관련 변경은 아래를 전부 통과해야 한다.

1. 영구 원본은 Firebase 사용자별 workspace 한 곳만 허용한다.
2. 브라우저 로컬 저장, 세션 저장, IndexedDB, 파일 캐시를 Studio 영구 원본으로 사용하지 않는다.
3. Firebase 성공 전에는 Studio 값에 의존하는 3D 화면·로비·Studio 편집기를 렌더하지 않는다. 로그인 UI와 타이틀 기본 UI는 막지 않는다.
4. 원격 경로가 비어 있다고 로컬값이나 기본값을 자동 업로드하지 않는다.
5. Firebase 실패 시 fallback하지 않고 치명적 오류로 중단한다.
6. 테스트에 `local-seeded`, offline local 유지 같은 성공 조건을 다시 만들지 않는다.
7. Studio 로컬 저장 접근 차단 테스트를 필수 회귀 테스트로 유지한다.
8. 코드 변경 후 실행 중인 개발 서버의 실제 HTTP 응답을 확인한다.
9. 실제 응답이 이전 코드면 즉시 서버를 재시작한다.
10. 파일 diff, 자동 테스트, 빌드, 서버 응답 네 가지가 모두 확인되기 전에는 복원 완료라고 보고하지 않는다.

## 9. 남은 사실과 한계

현재 코드는 Firebase 원격 스냅샷만 Studio 영구 원본으로 사용한다. 승인된 브라우저 저장 DB의 현재 후보를 최초 원격 스냅샷으로 직접 저장했고, Firebase에서 다시 읽은 데이터의 정규화 해시가 업로드 전 원본과 일치함을 확인했다.

로그인하지 않은 브라우저에서는 기존 Google 로그인 UI가 보인다. 로그인 후에만 사용자별 원격 경로를 읽고 Studio 의존 화면을 연다. 원격을 읽을 수 없는 경우 로컬 또는 기본 Studio 상태로 대체하지 않는다.

## 10. 최종 책임 결론

가장 큰 실패는 “Firebase에 연결했는가”만 확인하고 “Firebase만 원본인가”를 확인하지 않은 것이다.

직접 구현, 설계, 테스트 모두 같은 잘못된 로컬 우선 가정을 공유했고, 메인 세션이 이를 사용자 원문과 대조해 차단하지 못했다. 리뷰 경고도 집행되지 않았다. 오늘은 여기에 개발 서버 실응답 확인 지연이 겹쳐 사용자가 수정 후에도 구형 화면을 다시 보게 됐다.

이 사고는 사용자 명령의 모호함이 아니라 구현·검수·실행 확인의 연속 실패다.
