# Auto Deploy Stage 1 Loop / Leveling Stabilization Plan - 2026-06-24

프로젝트: Escape! zombie school
역할: Level_Mini / 레벨미니
범위: Stage 1 4분 모바일 플레이어블 루프, 레벨링, 난이도 곡선, 보상 페이싱, QA 핸드오프
상태: 기획/작업분해 산출물. 코드 변경 없음.

## 0. 결론

현재 우선순위는 신규 콘텐츠 확장이 아니라 Stage 1 240초(4분) 모바일 루프 안정화다. 이번 계획의 목표는 “모바일에서 실제로 조작 가능한 1판이 시작-성장-보스 조우-클리어/게임오버-보상 반영까지 끊기지 않는가”를 P0로 고정하고, 그 위에 레벨업/난이도/보상 수치를 QA가 검증 가능한 작업 단위로 쪼개는 것이다.

자동 배포 관점의 현재 판정은 조건부 No-Go다. gstack은 이번 실행에서 `GSTACK_OK`로 확인됐지만, 기존 QA 산출물 기준 실제 Android/WebView 1판 루프와 Playwright 모바일 스모크가 아직 완료되지 않았다. 또한 작업트리에 다른 에이전트/사용자 변경이 많으므로 이 문서는 코드 수정 없이 Planner 산출물로만 남긴다.

## 1. 정본 기준과 해석

### 1-1. 시간 기준

- 현행 Stage 1 길이: 240초 = 4분.
- 5분/300초가 남아 있는 문서는 2026-06-11 정책에 따라 시간값만 ×0.8로 환산한다.
- 보스 등장: 192초 = 3분 12초.
- 클리어: 240초 = 4분.
- 생존 마일스톤 골드: 48초 +1, 144초 +3, 192초 +4, 240초 +8.

### 1-2. 단위 기준

- 1 블록 = 4 units.
- Stage 1 코드상 현재 맵 반경: X ±14 units(±3.5 블록), Z ±54 units(±13.5 블록). `Bang_Rules.md`의 과거 정사각 96 units(24 블록) 기준과 다르므로, 실제 QA는 `stageConfig.js` 현행값을 우선 확인한다.
- 일반 적 스폰 링: 플레이어 기준 8.5-12.5 units(2.125-3.125 블록).
- 골드 폴백 드랍 링: 플레이어 기준 3.0-6.0 units(0.75-1.5 블록).
- Pencil 사거리: 22 units(5.5 블록).
- E05 돌진 감지 기준: 기존 문서 기준 4.5 units(1.125 블록). 코드/체감 QA에서 실제 동작 확인 필요.

### 1-3. Stage 1 적/난이도 기준

Stage 1은 접근/돌진 압박만 사용한다.

- 사용: E01, E02, E03, E05, E06, B01.
- 금지: E04 탄환형/원거리 적, B01 부채꼴 투사체.

## 2. 현재 확인된 제품 P0/P1을 Stage 1 루프로 재정렬

### P0-L1. 240초 1판 생존 루프 완결

목표:
- Stage 1이 모바일 환경에서 0초 시작 후 240초까지 타이머, 적 스폰, 자동 공격, XP/골드 수집, 레벨업, 보스, 클리어/게임오버 결과 화면까지 끊기지 않는다.

수락 기준:
- `STAGE_DURATION_SEC = 240`이 유지된다.
- 192초에 B01이 1회 등장한다.
- 240초 도달 시 보스 생존 여부와 무관하게 `cleared` 상태가 된다.
- HP 0 도달 시 `gameover` 상태가 되고 결과 화면에서 생존 시간/골드가 표시된다.
- Restart 후 Physics 트리 `gameKey` 리마운트와 `refs.js` 런타임 상태 리셋이 동작한다.
- 1판 중 JS uncaught error 0개.

QA 핸드오프:
- Balance_QA_Mini는 Web 스모크와 실제 Android/WebView 스모크를 분리 기록한다.
- 실제 Android/WebView에서 최소 1회는 자연 게임오버 또는 240초 클리어까지 진행한다.
- 앱 전환/백그라운드 복귀 뒤 입력이 stuck 되지 않는지 확인한다.

### P0-L2. 모바일 조작과 pause/resume

목표:
- 세로 모바일 화면에서 조이스틱 이동, 레벨업 카드 선택, pause/resume, restart가 손으로 조작 가능하다.

수락 기준:
- 390x844 뷰포트 또는 실제 Android 기기에서 조이스틱 이동 방향이 즉시 반영된다.
- pause 버튼 터치 후 적/무기/드랍/타이머가 멈춘다.
- resume 터치 후 동일 런이 이어진다.
- level-up 모달 카드 3개가 화면 밖으로 잘리지 않고 터치 가능하다.
- 결과 모달의 restart/title 계열 버튼이 터치 가능하다.

QA 핸드오프:
- 30초 이상 플레이 → level-up 카드 터치 → pause/resume → restart 순서를 한 세트로 기록한다.
- 실패 시 기기명, OS, 해상도, 입력 방식, 스크린샷/영상 경로를 남긴다.

### P0-L3. XP 연속 레벨업 처리

현재 상태:
- `useGameStore.gainXp()`는 while 루프로 여러 레벨업을 계산하고 `pendingLevelUps` 큐를 쌓는 구조다.
- 이전 역기획 문서의 “한 번 호출당 레벨업 1회만 처리” 위험은 현재 코드에서 일부 해소된 것으로 보인다.

목표:
- 엘리트/보스 보너스 교과서, XP 성장 패시브, 드랍 연속 수집으로 한 번에 여러 레벨업 조건을 넘겨도 카드 선택이 손실되지 않는다.

수락 기준:
- 한 번의 XP 획득으로 2레벨 이상 상승 가능한 테스트 케이스에서 `pendingLevelUps`가 누락 없이 증가한다.
- 카드 선택 1회 후 남은 pending이 있으면 다시 `levelup` 상태가 열린다.
- 카드 선택 중 게임 시간/적/무기 프레임이 진행되지 않는다.
- 240초 런에서 총 레벨업 수가 기록(`runLevelUps`)에 반영된다.

QA 핸드오프:
- 강제 XP 주입 테스트와 실제 플레이 드랍 테스트를 분리한다.
- “레벨은 올랐는데 카드 선택이 1번만 나왔다” 현상이 재현되면 P0 회귀로 분류한다.

### P0-L4. 보스 보상과 보상 손실 방지

현재 상태:
- B01 기본 XP는 0이지만 `getEliteBonusTextbookXp('B01', 0) === 40` 테스트가 존재한다.
- E06 보너스: 교과서 1개, 골드 1개.
- B01 보너스: 교과서 3개, 교과서 XP 40, 골드 5개.

목표:
- 보스/엘리트 처치가 보상 체감으로 이어지고, 골드가 세션/누적 모두에 반영된다.

수락 기준:
- E06 처치 시 교과서 1개와 골드 1개가 드랍된다.
- B01 처치 시 교과서 3개와 골드 5개가 드랍된다.
- B01 교과서 1개당 XP는 40으로 적용된다.
- 골드 획득 시 `goldSession`과 `school_survivor:goldTotal`이 증가한다.
- 클라우드 저장 실패가 결과 화면/코인상점/재시작을 막지 않는다.

QA 핸드오프:
- 보스 처치가 어려우면 QA/디버그 조건으로 B01 HP만 낮춘 별도 재현을 사용하되, 실제 밸런스 검증과 섞지 않는다.
- localStorage 주입값은 공식 플레이 보상 검증으로 표시하지 않는다.

관찰:

- 현재 코드 `weaponCatalog.js`는 starter 9종과 복원/신규 포함 총 14종에 가까운 카탈로그를 가진다.
- `upgrades.js`는 “Current game rule: max owned weapons per run is 8”로 되어 있고, 예전 문서의 4개 상한과 다르다.

이번 카드의 판단:
- 코드 수정은 하지 않는다.
- Stage 1 안정화 전에는 “무기 수 확장” 자체를 새 목표로 잡지 않는다.
- QA는 현행 코드 기준으로 카드 풀이 너무 넓어 초반 핵심 무기 성장이 흐려지는지 확인한다.

수락 기준:
- 240초 1판에서 카드 후보가 너무 분산되어 starter 핵심 무기 1개도 Lv.5에 못 가는 일이 반복되면 P1 밸런스 이슈로 올린다.
- Lv.2/Lv.4/Lv.6/Lv.8 카드 게이트가 실제 코드/문서와 불일치하면 “정본 선택 필요”로 Planner 후속 카드 생성.
- 신규/복원 무기 조건은 Stage 1 안정화 전에는 자동 배포 Go 조건에 포함하지 않는다.

### L2-003. 카드 선택 UI와 성장 손실 방지

수락 기준:
- 레벨업 시 카드 3개가 항상 보인다.
- 한 화면에 pencil 계열 카드가 과다하게 몰려 성장 선택이 단조로워지는지 확인한다.
- 활성 무기 Lv.5 도달 후 해당 무기 강화 카드가 더 이상 나오지 않는다.
- 보유 무기 상한에 도달한 뒤 acquire 카드가 나오지 않는다.
- `resumeFromLevelup()` 후 pending 레벨업 큐가 있으면 다음 카드 선택으로 이어진다.

QA 핸드오프:
- 카드 후보 30회 로그를 수집해 acquire/강화/패시브 분포를 기록한다.
- 모바일 터치 실패와 카드 로직 실패를 분리한다.

## 6. 자동 배포 전 Stage 1 Go / No-Go 게이트

Go 조건:
1. `npx vitest run --maxWorkers=1 --no-file-parallelism` 통과.
2. `npm run build` 통과.
3. 웹 스모크: 타이틀 → Stage 1 시작 → 30초 이상 생존 → level-up 카드 선택 → pause/resume → restart 통과.
4. 모바일 스모크: 390x844 또는 실제 Android에서 터치 이동, level-up 카드 터치, pause/resume, restart 통과.
5. Android WebView 또는 설치 빌드에서 자연 게임오버 또는 240초 클리어까지 1회 진행.
6. Stage 1 E04 wave/burst 제외 테스트 통과.
7. B01 Stage 1 투사체 없음 확인.
8. XP pending level-up 손실 없음 확인.
9. 보상 반영: 마일스톤/골드/보스 보너스/localStorage 누락 없음 확인.
10. 작업트리 변경 소유권과 QA 증거가 역할별 문서에 연결됨.

No-Go 조건:
- 실제 모바일 입력이 불가능하거나, pause/resume 후 입력이 stuck 됨.
- 1판 중 JS uncaught error, 반복 WebGL context lost, 검은 화면, 앱 재시작 필요 현상 발생.
- 240초 클리어/게임오버 결과가 표시되지 않음.
- XP/골드/패시브 저장이 플레이 루프를 막음.
- Stage 1에 E04 또는 투사체형 B01이 재등장.
- 현재 다수 미커밋 변경을 리뷰/검증 없이 자동 배포하려는 경우.
