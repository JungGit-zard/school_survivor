# Studio·게임·타이틀 캐릭터 절대법칙

이 문서는 Graphics Studio 캐릭터와 실제 게임 캐릭터와 타이틀 화면 캐릭터의 연결에 관한 프로젝트 절대법칙이다.

## 1. 절대 동일성

`Graphics Studio의 캐릭터 = 실제 게임의 캐릭터 = 타이틀 화면의 캐릭터`

세 화면의 캐릭터는 동일한 모델 원천, 동일한 모델 트리, 동일한 파츠 구성과 동일한 Firebase Studio 값을 사용한다. 별도의 타이틀 모델이나 비슷하게 다시 만든 모델은 존재할 수 없다.

## 2. 유일하게 허용되는 적용 순서

1. Firebase에 영구 저장된 현재 캐릭터 revision을 읽고 검증한다.
2. Firebase Studio의 루트, 파트, 그룹, 재질, 색상, 외곽선, 데칼 및 그 밖의 모든 캐릭터 값을 동일 모델에 하나도 누락하지 않고 적용한다.
3. 그 완성된 동일 캐릭터를 실제 게임과 타이틀이 함께 사용한다.
4. 타이틀에서만 필요한 위치, 회전, 스케일 및 애니메이션은 완성된 Studio 캐릭터 전체의 바깥쪽 연출 계층에 추가한다.

## 3. 즉시 적용

Studio에서 캐릭터를 변형하면 Firebase 저장 성공과 현재 revision 확인 직후 실제 게임과 타이틀이 같은 revision으로 함께 갱신되어야 한다.

Studio, 게임, 타이틀 중 어느 하나라도 다른 값이나 과거 revision을 표시하면 치명적인 버그다.

## 4. 타이틀 전용 연출의 허용 범위

타이틀 전용 위치, 회전, 스케일 및 애니메이션은 허용된다.

단, 이 허용은 Firebase Studio의 모든 값을 완벽하게 적용한 동일 캐릭터를 먼저 가져온 뒤 그 전체에 별도로 합성하는 경우에만 유효하다.

타이틀 전용 연출은 Studio 값을 대체·누락·초기화·복구·덮어쓰기할 수 없다. 타이틀 전용 연출을 제거하면 Studio와 게임의 캐릭터와 완전히 동일한 캐릭터가 남아야 한다.

## 5. 치명적인 버그

다음은 예외 없이 전부 치명적인 버그다.

- 타이틀 전용 캐릭터 모델
- 별도 메시, 복제 메시, 프록시 모델 또는 비슷하게 다시 만든 모델
- Firebase Studio 값을 적용하지 않은 로컬 원천 모델의 직접 렌더
- 로컬 시드, `localStorage`, 소스 기본값, 과거 revision 또는 임시 JSON의 사용
- 다른 브랜치, 작업공간, 아카이브 또는 빌드 산출물에서 변형값을 가져오는 행위
- Studio 값의 일부 누락
- 타이틀 전용 값으로 Studio 값을 대체하거나 덮어쓰는 행위
- Studio, 게임, 타이틀이 서로 다른 Firebase revision을 소비하는 상태
- 같은 컴포넌트나 파일을 import했다는 이유만으로 동일성을 검증 완료하는 행위
- 이 문서에 규정된 유일한 적용 순서 이외의 모든 과거 구현, 기록, 복구 방식, 연결 방식 및 방법론

## 6. 검증 법칙

검증은 파일명이나 컴포넌트 공유 여부가 아니라 다음 사실을 확인해야 한다.

- 세 화면이 동일한 Firebase revision을 소비하는가
- 모든 루트·파트·그룹과 시각 값이 누락 없이 적용되는가
- Studio 변경이 게임과 타이틀에 함께 즉시 반영되는가
- 타이틀 전용 연출을 제거했을 때 Studio·게임·타이틀 캐릭터가 완전히 동일한가

하나라도 충족하지 못하면 완료가 아니며 즉시 치명적인 버그로 판정한다.
