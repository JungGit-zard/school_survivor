## 판정: 조건부 가능 (CONDITIONAL GO)

**난이도 곡선 설계 자체는 타당하고 근거가 탄탄하다**(질적 상승 레버 구성, target 억제, 보스 배치 타이밍 모두 합리적). 그러나 설계가 전제한 **"신규 코드 0으로 성립"(D3)이 현재 스폰 엔진 기준으로 사실이 아니며**, 이 때문에 설계가 표방한 "3축 겹침" 정체성이 **코드 수정 없이는 구현되지 않는다.** 곡선은 승인, 배선 전제는 두 건의 블로커로 반려. 아래 2개 블로커 해소를 조건으로 구현 진행 가능.

블로커는 **난이도 수치 문제가 아니라 배선(wiring) 문제**다. levelmini의 phase/weight 테이블은 그대로 유효하다.

## BLOCKER-1 — STAGE3_BURST_EVENTS의 보스 외 항목은 현재 엔진에서 폐기된다 (사문화)

`Enemies.jsx` 471~483행, 유지 프레임 루프:
```js
// 보스(B01/B02) 등장만 버스트 스케줄에서 발화 — 나머지 버스트(형태/그룹)는 폐기(2026-07-11).
burstEvents.forEach((evt, idx) => {
  if (evt.type !== 'B01' && evt.type !== 'B02') return   // ← 보스 외 전부 무시
  ...
})
```

- 설계의 `STAGE3_BURST_EVENTS` 중 **실제로 발화하는 것은 `{sec:150,B01}`·`{sec:150,B02}` 두 줄뿐**이다.
- 폐기되는 것: 온보딩 초기밀도(`sec:0 E01×12`), 조기등장 신호(`E04×1`·`E05×2`·`E06×1·×1`), **그리고 formation 스파이크 4종 전부**(ring/pincer/swarm/gauntlet).
- `formationSpawnPositions()`는 이제 **테스트에서만 호출**된다(`Enemies.test.jsx`). 라이브 스폰 루프는 formation을 쓰지 않는다 → **"개방 아레나 360° 완전 포위(ring/gauntlet)" 시그니처는 코드상 실현 경로가 없다.** 현재 포위감은 기본 스폰 링(4.0~6.5, 360°)만으로 나온다.
- 설계 4-1의 주석 *"형태 버스트는 유지 루프가 총원을 target과 비교하므로 채운 만큼 자동으로 덜 스폰(의도된 상쇄)"*는 **stage2 구(舊) 동작 기준의 stale 진술**이다. 현재 이산 스케줄러는 target-대비-총원 비교를 **하지 않으며**(웨이브마다 `waveSize`만큼 무조건 추가), 애초에 보스 외 버스트가 발화하지 않으므로 이중으로 성립하지 않는다.

### 파급: E06(거대) 등장 보장 상실
설계는 `sec:108`·`sec:132` E06 버스트로 거대 조기 등장을 **보장**하려 했으나 이 버스트가 죽어서, E06 등장은 **오직 phase weight 확률에만 의존**한다. 표집 분산 분석(아래 (2c)) 결과 **~6~10% 런에서 거대를 한 번도 못 만난다.** 시그니처 몹의 조우 실패율로는 높다.

### 권고
1. `STAGE3_BURST_EVENTS`를 **보스 2줄로만 정리**하고, 나머지 신호/formation 줄은 문서에서 삭제하거나 "미발화(참고용)"으로 명시. `STAGE3_SPAWN_TELEGRAPHS`(포위/협공 예고 배너)도 대응 스폰이 없으므로 **그대로 두면 "예고만 뜨고 아무 일 없는" 허위 배너**가 된다 → 삭제.
2. formation 포위를 스테이지3 정체성으로 유지하려면 **스케줄러에 formation 재도입이 필요**(levelmini+threemini 스코프, D3를 "코드 필요"로 정정). 유지 시 halfX 12 개방맵 기준 `RING_MAX_RADIUS(5)`·`GAUNTLET_WALL_INSET` 재튜닝 병행.
3. E06 조우 보장을 위해 **weight 경로 강화**: 아래 (2c) 참조.

## BLOCKER-2 — E04 원거리 발사가 stage2에 하드코딩됨: 스테이지3 "원거리 축"이 비작동

`Enemy.jsx` 526행:
```js
const canFire = currentStageId === 'stage2' && canE04FireProjectile({ ... })
```
추가로 `canE04FireProjectile`(stage2ProjectileRules.js 23행)는 `elapsedSec < STAGE2_E04_INTRO_SEC(72)`를 내부에서 강제하고, E04 cap(`Enemies.jsx` 502행)·`getStage2E04Cap`도 `currentStageId==='stage2'` 게이트다.

**스테이지3에서 E04는:**
- `ranged:true`라서 kiting 거동(preferDist 5.5, minDist 3.5 유지·스트레이프)은 하지만 → **투사체를 절대 발사하지 않는다.**
- 즉 E04는 원거리에서 어슬렁대는 **접촉 데미지(8)만 있는 무해한 부유물**이 된다. 오히려 스폰 슬롯을 차지하며 압박을 **낮춘다.**
- 결과: 설계의 "서 있어도 죽고 돌아도 죽는" **3축(차저+거대+원거리) → 실질 2축(차저+거대)**으로 붕괴. 스테이지3의 핵심 정체성이 코드상 성립하지 않는다.

### 권고
E04를 스테이지3에서 살리려면 최소 3곳 수정 필요:
1. `Enemy.jsx` 526행: `currentStageId === 'stage2'` → 스테이지 파라미터화(stage3 허용).
2. `canE04FireProjectile`: `STAGE2_E04_INTRO_SEC(72)` 하드코딩을 stage별 intro(stage3=34)로 파라미터화. 함수/상수명도 stage-agnostic화 권장.
3. `Enemies.jsx` 502행 E04 cap 게이트: stage3 포함. **스테이지3는 E04를 조기·다구간 고비중 배치**하므로 cap 미적용 시 후반 원거리 과밀 위험 → cap을 stage3에도 적용하되 상한을 조정(예: `elapsedSec>=132`에서 3).

이 수정 없이는 D3("신규 코드 없이 성립") 전제가 거짓. **곡선을 그대로 쓰되 D3만 "E04 배선 코드 필요"로 정정**하면 된다.

## 검증 항목별 정량 결과

### 1. 난이도 곡선의 타당성 — 성립(승인)
- **질적 상승 근거는 수치로 성립**한다. 실효 HP: E06 256(스2)→320(스3, +25%), E02 56→70. 무기 마릿수(target 16~32)를 스2와 동급으로 억제하고 상승을 HP·조기도입·동시성으로 옮긴 판단은 **모바일 공정성 관점에서 옳다.**
- **조기 도입 사슬**(E04 34s·E05 52s·E06 108s)은 phase weight로 인코딩되어 있어 **버스트가 죽어도 곡선 자체는 유지**된다(BLOCKER-1이 곡선을 깨지 않는 이유).
- **온보딩 16s 압축**: 스1 40s/스2 24s 대비 공격적이나, "2스테이지 클리어 전제"이므로 **과하지 않음**. 단조 최소화 취지 타당.
- 상승이 마릿수가 아닌 질적 요소에서 나온다는 설계 의도는 target 환산표(피크 16마리 ≈ 스2 15마리 +1)로 확인됨.

### 2. 최우선 리스크 3종 정량

**(2a) 더블 보스 2300 HP — 처치보다 "90초 생존" 프레임이 정확**

보스 처치 압력(HP ÷ 창) 비교:
| 스테이지 | 보스 HP(실효) | 처치창 | HP/초 |
|---|---|---|---|
| 스1 | 1150 | 120→150 = 30s | 38.3 |
| 스2 | 920(×0.8) | 120→240 = 120s | 7.7 |
| **스3** | **2300** | **150→240 = 90s** | **25.6** |

- 스3 처치압력 25.6은 스1(38.3)과 스2(7.7) 사이, 스1에 가깝다. 스1이 클리어되는 게임이므로 **2스테이지 업글을 얹은 플레이어에게 25.6 DPS-on-boss는 달성 가능선**. 단 **트래시(보스구간 target 16~20) 동시 처리 시 보스 전담 DPS가 깎임** → 실측 필요.
- **주의(판정 전제)**: 포탈=클리어라면 보스 처치는 선택이고 2300 HP는 "90초간 차저 2기 회피" 위협이다. **클리어 조건이 처치 필수인지 balanceqa가 런타임 확인 필요**(D-후속). 대부분 서바이버 관례상 생존 프레임이 실제 바(bar).
- **동시 협공 돌진 회피 가능성**: B01·B02 동일 파라미터(warnDist 6.0, warnDuration 800ms, chargeSpeed 1.4, chargeDuration 2200ms). 둘 다 플레이어를 추격하므로 **동시에 warn(dist<6.0) 진입 확률 높음**. 다만 charge 방향은 warn 시점 플레이어 위치로 **고정**되므로:
  - 두 보스가 **정반대**에 있으면 두 돌진선이 같은 점을 관통 → **한 번의 수직 회피로 동시 회피 가능**(치명 아님).
  - 두 보스가 **~90° 직교**로 있으면 두 돌진선이 플레이어 지점에서 교차 → 한쪽 회피가 다른 쪽 레인으로 진입 → **클리핑 위험(22+22=44 데미지 버스트).**
  - 즉 "무조건 회피 불가"는 아니나 **직교 접근 시 회피 마진이 사라지는 상황적 즉사 위험**. 동일 tick 스폰(둘 다 sec150)이라 **warn/charge 사이클이 동기화**되어 반복적으로 겹칠 소지.
- **완화책**: 스폰 시각 **스태거**(B02 150, B01 162~165) → 사이클 위상차로 동시 돌진 빈도 급감. 자산 0, 데이터만 조정.

**(2b) 더블 보스 + 마틸다 3중 겹침(196~240) — 조건부 공정, 완화 권고**

- 마틸다: `damage = player.maxHp×3`(즉사), charger, chargeSpeed `player.speed×2.8`, warnDist 6.0(Enemies.jsx 398~411행). **접촉=즉사 추격자, 도망 전제 설계.**
- 스2도 이미 "보스(120 스폰) + 마틸다(180)" 2중 겹침이 최대 60초 존재. 스3는 여기에 **보스 1기 추가**한 확장.
- matildaSec 210(스 기본 180보다 늦춤)으로 창을 30초로 줄였으나, 보스 2기가 210까지 생존 시 **차저 3기(1기 즉사)** → 회피 레인 고갈 → **불공정 즉사 위험 구간**.
- **완화책(복합 권고)**:
  1. **보스 스폰 132~138s로 앞당김** → 처치창 102~108s(HP/초 21.3~22.5로 완화)로 넓혀 보스가 210 전 사망할 확률↑ → 3중 겹침 자체를 희석. (설계 리스크노트 1의 "132s 앞당김"과 정합)
  2. **matildaSec 210 → 218~222** → 3중 겹침 창을 18~22초로 축소.
  3. 둘 병행 시 "보스는 대체로 죽어 있고, 마틸다 겹침은 20초 이내"로 공정성 확보.

**(2c) 표집 분산 — E06 조우 실패율이 실질 위험**

20~40초 균등랜덤, 첫 웨이브 t=0, 평균 ~8웨이브. 활성 phase는 `findLast(start)`.
- **수학적 사실**: 연속 웨이브 간격 최대 40s이므로 **폭 ≥40s 구간에는 반드시 웨이브 1회 이상**이 떨어진다. E06 구간 108~150(폭 42s)은 **웨이브 발생 보장**됨.
- 그 웨이브의 waveSize ≈ 15(target 30~32), 구간 E06 weight 0.12~0.15 → `P(E06≥1) = 1-(0.86)^15 ≈ 0.88`.
- 백업 구간 172~196(폭 24s, weight 0.06)은 웨이브 미보장 + 저비중 → 방어력 약함.
- **종합 P(전 런 E06 미조우) ≈ 0.12 × (172~196 실패확률 ~0.5) ≈ 6~10%.** 시그니처 몹 조우 실패율로 높다. **버스트 보장(sec108/132)이 죽었기 때문**(BLOCKER-1).
- E04는 34~92·150~172(weight 0.18)·218~240 등 광범위 분포 → `P(전 런 미조우) < 2%`로 견고. **단 E04는 stage3에서 발사 안 함(BLOCKER-2)이라 조우해도 위협 없음** → 수정 전까지 E04 분산 논점은 무의미.
- **권고(E06 바닥 상향)**: (a) `150~172` phase에 E06 0.05~0.08 추가, (b) `172~196` E06 0.06→0.10 상향 중 택1로 미조우율을 ~2~3%로 낮춤. 버스트가 죽었으므로 weight로만 방어해야 함.

### 3. 모바일 공정성 / 렌더 부하 — 실측 조건부, HP×1.0이 진짜 드라이버

- **핵심 구조 사실: 동시 좀비 수 상한이 없다.** 스폰 루프는 웨이브마다 `waveSize`만 무조건 추가하고(Enemies.jsx 491~516), **target은 동시성 상한이 아니다.** 전역 enemy cap 부재(grep 확인). 동시성 = 스폰율 − 처치율의 누적.
- 피크 웨이브(132~150) waveSize 16은 스2 피크 15와 **+1 수준** → per-wave 스파이크는 스2와 사실상 동급.
- **진짜 부하 드라이버는 HP×1.0**: 전 적 TTK +25% → **정상상태 동시 개체수 ≈ 스2 × 1.25 + 보스 2기**(비인스턴스 메시, INSTANCED_TYPES 제외) + E06 상시화(320 HP·장수). 스2가 이미 모바일 예산 근처면 **스3는 초과 가능.**
- **판정**: per-wave는 안전, **누적 동시성은 실측 필요**(balanceqa 런타임, 저사양 기기 프레임/개체수 카운터). 대책이 필요하면 (4)의 부분 완화가 동시성도 함께 낮춘다.

### 4. HP ×1.0 정책 — 방향 타당, 부분 완화를 조건부 권고

- 스2 ×0.8 → 스3 ×1.0(+25%)은 설계의 주 상승 레버로 **정당**. 2스테이지 카드 업글은 통상 +25% 이상 DPS를 주므로 **상대 난이도는 관리 가능**.
- 단 side-effect가 **체감보다 동시성/그라인드**: TTK↑ → 사체 누적↑(3항), E06(320)+E02(70) 고비중 겹치는 108~150에서 "잡아도 안 줄어드는" 스택.
- **권고**: (3)의 동시성 실측이 예산 초과로 나오면 **잡몹만 부분 완화**(E01/E03 ×0.9, E02/E05/E06/보스 ×1.0). 채프 클리어는 스냅하게 유지해 사체 누적을 줄이면서, 무거운 위협의 탱키함(정체성)은 보존. 실측 이전에는 ×1.0 유지로 착수해도 무방(override 미생성 = 자동 ×1.0).
