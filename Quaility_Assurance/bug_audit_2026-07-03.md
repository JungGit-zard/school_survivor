- 심각도: High
- 카테고리: Release / Functional / Security-ish

### 재현 단계

1. `Developer/r3f_prototype`에서 `npm run build` 실행
2. `npm run preview -- --host 127.0.0.1 --port 4173` 실행
3. `http://127.0.0.1:4173/` 접속
4. 우상단 `치트` 버튼 확인
5. `치트` 클릭

### 기대 동작

- Stage 선택, 모든 무기 해금, 코인 레벨업 초기화 같은 개발 기능은 일반 테스트 빌드에서 기본 비노출이어야 함.

### 실제 동작

- 타이틀 화면 우상단에 `치트` 버튼이 노출됨.
- 치트 메뉴에서 다음 기능에 접근 가능:
  - Stage 1 / Stage 2 시작 선택
  - 모든 무기 해금
  - 코인 레벨업 초기화

### 참고 코드 근거

- `src/lib/adminConfig.js`
  - `DEFAULT_ADMIN_CONFIG.operations.cheatMenuButtonVisible: true`
- `src/components/TitleScreen.jsx`
  - `cheatMenuButtonVisible`가 true면 치트 버튼/모달 렌더

### 권장 조치

- 릴리스 기본값은 `cheatMenuButtonVisible: false`로 변경.
- 개발 빌드에서만 열어야 한다면 `import.meta.env.DEV` 또는 명시적인 QA 플래그로 보호.
- 비공개 테스트용 빌드에서도 치트 노출이 의도라면, 최소한 `외부 테스터용 QA 빌드`와 `릴리스 후보 빌드`를 분리.

## 2. 게임오버/클리어 결과 화면에 개발 로그 복사 버튼이 노출될 수 있음

- 심각도: High
- 카테고리: Release / UX / Debug leakage
- 수동 관찰 위치:
  - 게임오버 결과 화면 우측에 `개발 로그 복사` 버튼 노출 확인

### 재현 단계

1. 게임 시작
2. 사망하여 게임오버 결과 화면 진입
3. 결과 화면 우측 확인

### 기대 동작

- 일반 유저/외부 테스터 대상 결과 화면에서는 개발 로그 복사 버튼이 보이지 않아야 함.

### 실제 동작

- `개발 로그 복사` 버튼이 결과 화면의 일반 액션과 함께 노출될 수 있음.

### 참고 코드 근거

- `src/components/HUD.jsx`
  - `showResultDevTools = getAdminOperationsConfig().cheatMenuButtonVisible && (phase === 'gameover' || phase === 'cleared')`
- 현재 기본 admin config가 치트 버튼 노출 true이므로 결과 개발도구도 기본 노출 조건을 만족함.

### 권장 조치

- 결과 개발도구는 `import.meta.env.DEV` 또는 별도 `playtestLogToolsVisible` 플래그로 분리.
- `cheatMenuButtonVisible` 하나로 타이틀 치트와 결과 개발 로그를 동시에 제어하지 않는 편이 안전함.

## 3. 레벨업 선택 직후 적에게 둘러싸인 상태에서 즉시 사망 가능성이 큼

- 심각도: Medium
- 카테고리: Gameplay / UX
- 수동 관찰:
  - Lv.2/Lv.3 레벨업 선택지가 뜬 시점에 플레이어 주변에 적이 겹쳐 있었음.
  - 선택 중 시간은 멈춘 것으로 보였으나, 선택 완료 직후 다수 적과 겹친 상태에서 게임오버로 이어짐.

### 재현 단계

1. Stage 1 시작
2. 적에게 둘러싸인 상태에서 경험치 획득으로 레벨업 진입
3. 선택지 중 하나를 선택
4. 선택 직후 플레이어 생존 여부 확인

### 기대 동작

- 레벨업 선택 직후 최소한의 안전장치가 있어야 함.
  - 짧은 무적 시간
  - 적 밀어내기
  - 플레이어 위치 복구
  - 선택 중 접촉 피해 완전 정지 및 선택 후 짧은 grace period

### 실제 동작

- 적이 플레이어와 겹친 상태에서 선택이 끝나면 즉시 피해/사망으로 이어질 수 있음.

### 권장 조치

- 레벨업 종료 후 0.5~1.0초 정도 무적 또는 knockback 제공 검토.
- 레벨업 오버레이가 뜰 때 적과 플레이어의 겹침 상태를 해소하거나, 선택 완료 직후 첫 충돌 피해를 무시.

## 4. 랭킹 화면 하단 고정 버튼이 랭킹 리스트 하단 영역을 덮음

- 심각도: Medium
- 카테고리: UI / Layout
- 근거 스크린샷:
  - `screenshots/bug_audit_2026-07-03/03_ranking_bottom_button_overlap.png`

### 재현 단계

1. 타이틀 화면에서 `유저랭킹` 클릭
2. 랭킹 화면 하단 확인

### 기대 동작

- `타이틀로 돌아가기` 버튼이 리스트 내용을 가리지 않아야 함.
- 리스트는 하단 버튼 높이만큼 padding 또는 safe area를 확보해야 함.

### 실제 동작

- 하단의 큰 `타이틀로 돌아가기` 버튼이 랭킹 리스트의 아래쪽 행을 덮는 형태로 보임.

### 권장 조치

- 랭킹 리스트 컨테이너 하단에 버튼 높이 + 여백만큼 padding 추가.
- 또는 하단 버튼을 sticky footer로 두되 리스트 영역 높이를 명확히 계산.

## 8. 안정화 지침 기준: `useFrame` 내부 React state 업데이트가 남아 있음

- 심각도: High
- 카테고리: Performance / R3F-Rapier stability
- 기준 문서:
  - `Developer/agent_room/r3f_rapier_vampire_survivor_stability_rules.md`
  - 위반 기준: RULE-0.1, RULE-0.3, RULE-4.1
- 코드 근거:
  - `Developer/r3f_prototype/src/components/Enemy.jsx:344` `useFrame((_, delta) => { ... })`
  - `Enemy.jsx:414` `setProjectiles((prev) => [...prev, { ... }])`
  - `Enemy.jsx:441`, `451`, `466`, `472` 등 `setAnimPhase(...)`

### 재현/확인 단계

1. 안정화 지침 정본의 Golden Rules 확인
2. `Enemy.jsx`의 `useFrame` 루프 확인
3. ranged enemy 투사체 발사/charger 상태 전환 지점에서 React state setter 호출 확인

### 기대 동작

- 프레임 루프에서는 React state를 직접 갱신하지 않는다.
- 투사체/애니메이션 상태는 풀링된 mutable store/ref/SoA 구조로 갱신하고, UI에 필요한 값만 저빈도 동기화한다.

### 실제 동작

- ranged enemy 발사 시 React state 배열을 새로 만들며 투사체를 추가한다.
- charger 상태 전환 시 프레임 루프 안에서 애니메이션 state setter를 호출한다.

### 영향

- 몬스터 수가 늘거나 Stage 2 원거리 적이 많아질수록 리렌더/GC/히치 위험이 커진다.
- 안정화 지침상 “느려짐 / 물리 이상 / 사라짐” 계열의 1차 위험 패턴이다.

### 권장 조치

- 투사체는 최대 수량 풀 + active flag로 전환한다.
- `setProjectiles` 대신 projectile pool ref/store에 write하고 렌더는 InstancedMesh/registry에서 읽게 한다.
- `setAnimPhase`는 전환 이벤트를 ref에 기록하거나 visual registry field로 관리한다.
- React 상태가 꼭 필요하면 프레임 밖 큐에 쌓고 throttle된 단일 commit으로 제한한다.
